# 第一部分：项目介绍

## 1.1 问题背景与现实痛点

各位老师、同学，下午好。我们今天汇报的项目是"PPT 大纲智能生成与内容补全系统"。

在企业汇报、学术展示或内部沟通等典型场景中，一份称得上"高质量"的 PPT，通常需要经历以下几个环节：
首先是对主题进行系统性分析，厘清整体的汇报逻辑；
其次是从多源渠道检索、筛选并核实相关信息；
然后是构建合理的章节结构与叙事顺序；
在此基础上，还需要将复杂的信息提炼为适合单页展示的精炼要点；
最后，往往还需要根据反馈进行多轮迭代修改。

这一整套流程，少则数小时，多则跨越数天。而对于缺乏经验的用户而言，"不知道从哪里开始"本身往往就是最大的认知障碍。

---

## 1.2 现有 AI PPT 工具的局限性

面对这一需求，市场上已涌现出一批 AI PPT 生成工具。然而，经过调研后我们发现，这些工具普遍存在几个难以回避的核心缺陷。

第一，内容可靠性问题。现有工具大多依赖纯大语言模型直接生成内容，缺乏真实数据与外部知识的支撑，容易产生"看起来正确、实际上空洞"的内容——语言流畅，但数据失真。

第二，信息时效性问题。模型的训练知识存在截止日期，无法反映当下最新的行业动态与研究进展。对于需要引用最新数据的汇报场景而言，这是一个根本性的短板。

第三，结构灵活性问题。现有工具大多采用固化模板，无法根据具体的汇报场景、目标受众和内容主题进行动态调整，生成结果千篇一律。

第四，私有知识利用问题。企业和研究人员在实际使用中往往持有大量私有文档——调研报告、会议纪要、内部数据——这些信息现有工具无从利用。

第五，交互深度问题。现有工具通常是一次性生成，用户无法在同一会话中持续迭代、精细化调整，生成结果与实际需求之间的距离往往难以弥合。

---

## 1.3 系统定位与核心目标

正是基于对上述问题的系统性分析，我们确立了本系统的定位，面向内容深度生成的智能 PPT 创作辅助系统，而非简单的"一键生成"工具。

本系统的核心目标，是通过融合大语言模型的内容生成能力与外部知识增强机制，将用户输入的简短主题描述或长篇文档，稳定转译为结构清晰、内容可靠、有数据支撑的 PPT 草稿。

在成功标准上，我们从四个维度进行界定。在效率维度，目标是帮助用户将 PPT 制作时间降低至少 50%；在质量维度，大纲结构合理性通过率应达到 90% 以上，内容准确性与信息相关性相较纯大语言模型方案应有显著提升；在体验维度，用户体验评分目标为 4 分及以上（满分 5 分，基于实验评估）；在产品维度，系统需具备一定的产品化潜力，可集成于智能办公工具链。

---

## 1.4 两种核心创作路径

为了覆盖用户在实际使用中的两类典型起点，本系统设计了两条核心创作路径。

第一条是报告驱动模式（REPORT_DRIVEN）。这一模式适用于用户已持有完整文档的场景——例如研究报告、市场调研数据或会议纪要。用户在发送首条消息时可以选择携带报告文件，系统基于此自动识别模式类型，完成文档解析与文本提取后，跳过需求收集阶段，直接进入大纲生成流程。在大纲生成和内容生成阶段，报告文本将作为作为主要依据，使生成内容始终紧扣原始文档。

第二条是引导式创作模式（GUIDED）。这一模式针对用户仅有一个模糊主题想法的场景，也是我们认为更具普适价值的核心场景。用户仅需发送纯文字主题，系统识别模式后进入多轮对话，逐步收集与汇报主题、目标受众、内容时长、展示风格、重点内容等相关的需求信息。需求完整后，系统触发大纲生成，并在内容生成阶段同时接入 RAG 检索与 DeepSearch 实时网络搜索，双路并行增强内容质量。

无论采用哪种路径，整个创作过程都被约束在统一的六阶段状态机框架之内：需求收集、大纲生成中、大纲确认与修改、内容生成中、内容确认与修改，直至最终完成。所有阶段状态均持久化至数据库，支持断点恢复与多会话并发。

---

## 1.5 技术栈概览

在技术选型上，我们围绕上述核心需求构建了一套五层技术架构。

前端采用 React、TypeScript 与 Vite 的组合，负责对话界面、大纲编辑器与 SSE 流式内容展示。后端框架选用 FastAPI，基于 Python 3.11 全异步实现，承载 API 网关、业务逻辑与 SSE 长连接维护。

数据存储方面，我们选用 PostgreSQL 作为核心关系数据库，并集成 pgvector 扩展，在同一数据库内同时支持业务数据与文档向量的一体化存储。异步消息队列采用 Redis Stream，用于驱动知识库文件的后台处理流程。原始文件的持久化存储则通过兼容 S3 协议的对象存储服务实现。

在 AI 能力层，大语言模型通过标准化兼容接口接入，支持多服务商可配置，用户可根据需要自行配置 API Key。值得特别说明的是，RAG 的文档向量化模型固定采用阿里云 DashScope 的 text-embedding-v3，这一决策是有意为之：固定 Embedding 模型确保了同一用户在历史向量与新增向量之间的维度始终一致，从根本上规避了因模型切换导致检索失效的风险。向量检索通过 LlamaIndex 与 pgvector 的集成实现，采用余弦相似度，支持 Top-K 段落召回。AI 流程编排则选用 LangChain 与 LangGraph 的组合，负责 RAG 流程调度与多轮检索。

整个开发环境通过 Docker 实现一键部署。

# 第二部分：项目 RBS（需求分解结构）

## 2.1 RBS 概述

接下来我们介绍项目的需求分解结构。

本项目的需求可以分解为三个主要方向：结构化大纲生成、知识检索与内容补全，以及端到端评估系统。这三个方向构成了本项目的三条核心研究与工程主线。

---

## 2.2 结构化大纲生成

第一个方向是结构化大纲生成模块。这一模块的目标是将用户输入的主题描述或文档内容，转化为具有四级叙事结构的 PPT 骨架——即章节、页面、要点与备注。

在具体的研究路径上，我们规划了三项核心工作。其一，Prompt 工程策略研究：系统性设计并对比多种提示工程策略，包括思维链提示、少样本学习等，通过量化指标筛选在大纲生成任务上表现最稳定的方案。其二，结构化输出约束设计：以 JSON Schema 对大语言模型的输出格式加以约束，确保任何模型的大纲输出始终符合预定义的层级规范，保障下游处理流程的稳定性，防止因模型输出格式不一致导致的解析失败。其三，多模型横向对比：在大纲生成任务上对多款主流大语言模型进行生成稳定性、输出质量与响应效率的系统性横向比较，以数据驱动技术选型。

---

## 2.3 知识检索与内容补全

第二个方向是知识检索与内容补全模块。这一模块的目标是为每一张幻灯片自动检索、过滤并压缩外部知识，将高质量的文本、数据与论据注入内容生成流程。

具体拆解为三项工作。RAG 知识检索模块：以大纲节点——即页面主题与核心要点——为检索锚点，将其自动改写为查询语句，通过向量相似度搜索在私有知识库中召回相关段落。DeepSearch 集成：接入网络搜索 API，实现多轮查询改写与跨源信息聚合，为知识库未覆盖的内容提供实时互联网信息补充，解决私有知识库覆盖范围有限的问题。内容压缩与合成：将检索到的长文本智能压缩为符合单页 PPT 展示约束的精炼要点，提取关键数据、核心观点与支撑证据，而非简单堆砌原文。

---

## 2.4 端到端评估系统

第三个方向是端到端评估系统。这一模块的价值在于为前两个方向的研究成果提供客观、可量化的质量基准。

评估维度涵盖结构合理性（章节组织与幻灯片粒度是否合理）、信息密度（每页信息量是否适当）与事实准确性（正确事实陈述的比例）。评估策略采用人工评估与自动化指标双轨并行：人工评估侧重内容的清晰度、完整性与展示就绪度；自动化评估则采用 ROUGE、BERTScore 等 NLP 指标，结合 PPT 专项规则评分。

---

# 第三部分：项目 WBS（工作分解结构）

## 3.1 WBS 概述

第三部分我们介绍工作分解结构，即 WBS。工作分解结构是将项目拆解为可管理工作单元的系统性方法，其核心工作领域均源自需求分解结构的最底层需求。本项目 WBS 涵盖三大核心工作领域：**结构化大纲生成**、**知识检索与内容补全**，以及**端到端评估系统**。在此之前，以**项目规划与设计**作为前置基础阶段，总预算工时 220 小时，以下分领域逐一介绍。

---

## 3.2 项目规划与设计（已完成，40h）

项目规划与设计阶段已全部完成，总投入 40 小时。需求收集与技术可行性分析由全体成员共同参与，历时约半周，对基于大语言模型的内容生成、RAG 技术与 DeepResearch 方法进行文献调研，明确项目目标与核心功能边界。系统设计阶段完成了整体架构确定、技术栈选型、API 接口规范制定与数据流管道设计；同期完成了 12 张核心数据库表的 Schema 建模，以及四级 PPT 大纲 JSON Schema 规范定义（章节→页面→要点→备注），为后续结构化输出约束奠定统一规范基础。

---

## 3.3 结构化大纲生成（进行中）

第一个核心工作领域是结构化大纲生成，对应 RBS 中结构化大纲生成方向的最底层需求，下设三个工作方向。

**Prompt 工程策略研究**

本工作方向将系统研究各类 Prompt 工程策略对大纲质量的影响，设计并比较不同提示方法，以识别能够生成逻辑连贯、结构合理 PPT 大纲的最优策略。具体工作包括三项：第一，设计思维链（Chain-of-Thought）提示模板，分别针对章节划分逻辑、页面粒度把控与叙事连贯性三个维度进行专项设计；第二，构建高质量 PPT 大纲少样本示例库，覆盖商业汇报、学术展示、项目介绍等多类场景；第三，实现 Prompt 策略对比专项测试框架，量化评估各策略在结构合理性与格式规范性上的表现差异。本工作方向设置评审节点，通过质量指标分析评估各 Prompt 有效性并筛选最优策略；测试节点则跨多类主题类别进行系统性覆盖验证。结合更新后的项目计划，本方向以子任务 2.2.2（预算 10 小时）推进，当前处于进行中状态。

**结构化输出约束设计**

本工作方向将基于 JSON Schema 设计并实现结构化输出约束，确保生成大纲严格符合预定义的四级层级规范，在逻辑连贯性、层级合理性与页面粒度三个维度维持一致质量。具体包括：完整定义四级大纲 JSON Schema，声明章节、页面、要点、备注的类型约束、数量约束与必填字段；实现 Schema 验证与自动纠错机制，对不合规输出启动自动修正或重试流程；针对内容极短或高度专业化的边界情况，设计动态约束松弛策略。目前四级大纲 JSON Schema 定义已完成，Schema 验证机制正在集成至大纲生成后处理流水线中。评审节点统计 Schema 合规率并识别典型失败模式，测试节点跨不同输入类型验证约束有效性。

**多模型横向对比**

本工作方向将对多家主流大语言模型服务商在大纲生成任务上进行全面横向对比，评估各模型在生成稳定性、输出质量与响应效率方面的综合表现。主要工作包括：为各服务商配置标准化 API 调用接口；设计统一评估基准与变量受控的测试数据集，覆盖多类主题领域；以控制变量法执行横向对比实验，并对模型性能差异进行统计显著性检验。结合更新后的项目计划，本方向以子任务 2.2.3（预算 12 小时）推进，计划于近期启动。评审节点分析对比结果以识别最优模型配置，测试节点验证统计显著性。

## 3.4 知识检索与内容补全（进行中）

第二个核心工作领域是知识检索与内容补全，同样下设三个工作方向，协同为每个 PPT 页面自动补全外部知识，提升内容的信息密度与事实可靠性。

**RAG 知识检索模块**

本工作方向将开发以大纲节点为锚点的查询改写策略，通过迭代检索自动获取与每个页面内容高度相关的外部信息。具体包括四项工作：构建基于向量 Embedding 的知识库索引基础设施，以 pgvector 独立命名空间隔离各文件向量数据；设计以大纲节点（页面主题与核心要点）为锚点的查询改写算法，将大纲节点自动转换为适合向量检索的查询语句；实现带相关性评分的多轮检索管道，采用余弦相似度 Top-K 召回；开发查询扩展与精炼机制，提升低相关性查询场景下的召回质量。评审节点跨不同查询类型评估检索精确率与召回率，测试节点结合标注数据进行端到端检索质量评估。结合更新后的项目计划，知识库索引基础设施已完成，pgvector 向量索引与查询改写算法（子任务 2.3.2，预算 15 小时）当前处于进行中状态。

**DeepResearch 集成**

本工作方向将集成 DeepResearch 能力，支持多轮查询改写与跨源信息整合，通过迭代搜索与内容分析逐步完善知识结构。主要工作包括：设计多轮搜索编排工作流，支持多服务商网络搜索 API 的可配置接入；实现采用 RRF（倒数排名融合）策略的跨源信息聚合与去重，将 RAG 检索结果与网络检索结果有机融合；开发基于中间检索结果的迭代查询精炼机制，逐步完善知识覆盖结构。评审节点评估检索信息的覆盖深度与广度，测试节点对比 DeepResearch 增强检索与单轮基线的效果差异。结合更新后的项目计划，本方向计划于 4 月 13 日至 5 月 11 日期间推进。

**内容压缩与合成**

本工作方向将开发智能内容压缩与合成方法，将大量检索文本压缩至单个 PPT 页面的展示约束内，提取关键数据、提炼核心观点并筛选支撑证据。具体工作包括：设计针对 PPT 单页展示约束的内容摘要流水线，从检索段落中抽取关键数据、核心观点与支撑证据；实现关键信息抽取与证据过滤算法，确保补全内容信息密度适当、不堆砌原文；开发页面级内容组装与连贯性优化机制，保证单页要点之间的逻辑衔接与表达规范性。评审节点通过信息保留率指标评估压缩质量，测试节点评估压缩内容的可读性与展示适用性。结合更新后的项目计划，本方向计划于 4 月 20 日至 5 月 11 日期间推进。

## 3.5 端到端评估系统（待开始）

第三个核心工作领域是端到端评估系统，计划于核心模块基本成型后启动，同样下设三个工作方向，从主观与客观两个维度对系统输出进行全面质量验证。

**评估指标设计**

本工作方向将建立一套涵盖结构合理性、信息密度与事实准确性三个维度的实用评估指标体系，为系统输出质量提供系统化、可量化的评估框架。结构合理性指标具体评价章节组织逻辑合理性与幻灯片粒度适当性；信息密度评分标准基于内容覆盖度与简洁性，量化每页信息量的充实程度；事实准确性核查程序基于参考资料对生成内容进行轻量级事实陈述核验。评审节点通过重复自评或小规模同伴评审检验指标一致性与可操作性，测试节点对基准样本应用评估指标进行初步校准，确保指标区分度合理。

**人工评估**

本工作方向将设计并执行小规模人工评估流程，借助同伴反馈获取对生成内容的主观质量评价，弥补自动化指标在语义理解层面的固有局限。具体包括：设计结构化评估问卷与评分标准，涵盖大纲结构合理性、内容相关性、表达专业性等核心维度；邀请具备基础 PPT 制作经验的同学担任评估人，确保评估背景与目标用户群体一致；以随机样本呈现顺序执行简单盲评实验，排除评估人对生成来源的先验偏见。评审节点分析评估人之间的评分一致性，识别主观分歧较大的评价维度；测试节点跨不同输出样本对比评估结果，验证评估流程的稳定性与可重复性。

**自动化评估**

本工作方向将利用标准 NLP 指标与 PPT 专项启发式规则实现自动化评估，确保评估结果的客观性、可复现性与批量处理能力。主要工作包括：实现 ROUGE、BERTScore 等标准自动化 NLP 评估指标，支持与参考文本的语义相似度度量；设计面向 PPT 专项特征的规则评分模块，覆盖长度控制合理性、关键词覆盖率、层级格式规范性等维度；构建支持批量处理的自动化评估流水线，实现对大批量生成样本的高效评分与统计汇总。评审节点分析自动化评分与人工评估结果的相关性以验证指标有效性，测试节点跨不同输入分布评估自动化指标的鲁棒性并修正边界情况评分偏差。结合更新后的项目计划，本领域整体计划于 2026 年 5 月 4 日至 5 月 18 日推进。

## 5.2 系统整体架构设计

在系统架构方面，我们确立并落地了一套五层架构体系。

最上层为前端展示层，由 React、TypeScript 与 Vite 构建，负责对话界面、大纲编辑器与 内容预览与知识库展示。
第二层是 API 网关层，由 FastAPI 承载，提供路由分发、JWT 鉴权、全局异常处理、请求日志等功能。
第三层是业务服务层，包含 SessionService、KnowledgeBaseService 与 ReportService 三个核心服务模块，分别负责会话全生命周期管理、知识库文件处理与报告解析服务。
第四层是核心能力层，涵盖 LLM 服务、RAG 服务与 DeepSearch 服务三个能力单元，为业务服务层提供 AI 能力支撑。
最底层是存储层，由 PostgreSQL 加 pgvector 承担关系数据与向量数据的一体化存储，Redis Stream 承担知识库文件的异步处理队列，兼容 S3 协议的对象存储服务承担原始文件的持久化。

本架构中有几个关键决策。

其一是延迟会话创建。传统设计中用户点击"新建会话"时即触发 API 创建会话记录，容易产生大量用户最终放弃的无效会话。我们的设计是，前端在用户点击"新建"时仅在本地展示输入界面，待用户发送第一条消息时，才通过 `POST /sessions/start` 接口原子化地完成会话创建与首条消息处理，消除无效会话的积累。

其二是 Prompt 外化。所有的 Prompt 模板文件与 JSON Schema 文件均独立存储于 `resources/` 目录中，运行时由 PromptLoader 动态加载，不与业务代码耦合。使 Prompt 可以独立调优与迭代，无需修改任何业务逻辑代码，降低提示工程的迭代成本。

其三是大语言模型多服务商适配。通过标准化兼容接口统一接入多个主流大语言模型服务商，用户可独立配置 API Key，有用户自己选择使用的模型服务商，并配置相应的 API Key，同时方便比较不同大模型的性能差异。

用户体系方面，`users` 表存储用户基础信息，包含角色区分与账号状态；`user_llm_configs` 表管理用户的大语言模型配置，支持多服务商，设有唯一的默认配置约束，API Key 通过 Fernet 对称加密存储；`user_rag_configs` 表每用户唯一，Embedding 模型固定，存储txt_embedding-v3 模型的 API Key；`user_search_configs` 表管理用户的搜索服务配置，支持多服务商与唯一默认配置约束。

## 5.5 Session / Task 异步状态机

状态机设计是本系统最具工程价值的核心设计之一。

在会话阶段状态机设计上，我们定义了六个枚举阶段：需求收集（REQUIREMENT_COLLECTION）、大纲生成中（OUTLINE_GENERATION）、大纲修改/确认（OUTLINE_CONFIRMING）、内容生成中（CONTENT_GENERATION）、内容修改/确认（CONTENT_CONFIRMING），以及已完成（COMPLETED）。这六个阶段构成严格的单向流转链路，不允许跳跃或回退。

这一设计带来了三项具体的工程价值。
第一，防止操作混乱，阶段的单向约束确保用户无法在内容生成阶段回退修改大纲结构，避免了状态不一致导致的数据损坏。
第二，支持断点恢复，由于所有阶段状态均持久化至数据库，用户在任意时刻断开连接后重新连接，系统可自动识别当前阶段并恢复上次进度。
第三，支持多会话并发，用户可以同时维护多个不同主题的 PPT 创作会话，各会话状态相互独立，互不干扰。

在任务类型设计上，我们主要定义了三类异步任务。
大纲生成任务（OUTLINE_GENERATION）在用户确认需求或直接触发时创建，以流式方式输出大纲内容。
大纲修改任务（OUTLINE_MODIFICATION）在用户于确认阶段提出修改需求时创建，支持局部或全量修改。
幻灯片批量生成任务（SLIDE_BATCH）在用户确认大纲后创建，系统为每一张幻灯片同时创建独立的子任务并行执行，使整体内容生成时间不随幻灯片数量线性增长，显著提升生成效率。

目前，Session Service 的七个核心方法接口骨架已完成定义，包括创建会话与处理首条消息、统一消息入口、需求收集处理、大纲生成处理、大纲确认与修改处理、内容生成处理、内容确认与修改处理，以及同步语义意图判断。各方法的业务逻辑实现正在推进中。

## 5.6 Prompt 工程与 JSON Schema 规范

在 Prompt 工程方面，本阶段取得了两项重要进展。

第一项是四级 PPT 大纲 JSON Schema 的锁定。Schema 定义了四个层级：第一级为章节（Chapter），构成全文骨架，通常 3 至 5 个；第二级为页面（Slide），每章 2 至 4 页；第三级为要点（Points），每页 3 至 5 个核心要点；第四级为备注（Notes），提供每页的补充说明与讲解思路。

Schema 通过 `required` 字段、类型约束，以及 `minItems` 与 `maxItems` 范围限制，对大语言模型的输出格式加以强制约束，确保任何模型的输出均可被可靠地解析为预期的层级结构，从而保障前端渲染与后续业务处理的稳定性。

第二项是意图识别 Prompt的初步验证。
这一 Prompt 在整个系统中扮演"决策门控"的角色，承担三项判断职责：
第一，判断用户消息是否与 PPT 创作无关，若是则返回引导性提示，不触发后台任务；
第二，判断用户是否有进入下一阶段（如明确表达"直接生成"等）意图，若是则直接进入下一阶段，如从需求收集阶段直接转到大纲生成阶段，或从大纲确认阶段直接转成内容检索、填充阶段；
第三，如果为用户需求、要求，则进入该阶段后续流程，如需求填充、大纲修改等。
这三项判断均输出结构化 JSON，供 Session Service 在同步处理路径中快速决策。

在 Prompt 文件整体规划上，共规划了七个核心文件。`intent_classifier.txt` 已完成初版并通过验证；`requirement_collector.txt` 与 `outline_from_requirements.txt` 处于草稿迭代阶段；`outline_from_report.txt`、`outline_modification.txt`、`slide_batch.txt` 与 `slide_target_classifier.txt` 待开发启动。

## 5.7 RAG 基础设施建设

在 RAG 基础设施方面，本阶段完成了知识库文件异步处理流程的搭建，并完整实现了知识库服务层的全部接口。

在文件处理流程上，用户上传文件后，系统首先计算文件的 SHA-256 哈希值，与该用户现有文件进行比对，若同内容文件已存在，则直接复用现有向量，不重复执行提取与向量化，节省了计算资源与存储空间。
若为新文件，则依次完成写入对象存储、创建 DocumentFile 记录、向 Redis Stream 推送处理消息三个步骤，上传请求不阻塞等待后续处理。

后台的 KnowledgeWorker 以 Consumer Group 方式消费 Redis Stream 消息，按文件格式分别调用对应的解析器，提取纯文本后执行语义分块，批量调用 DashScope Embedding API 完成向量化，将向量存入 pgvector。处理成功后将文件状态更新为 READY，处理失败则标记为 FAILED 并保留重试入口。

知识库服务层已实现文件上传、文件列表查询、单文件状态查询、失败重试、文件删除，以及会话与知识库文件多对多引用的批量增加、删除与查询等接口。

pgvector 向量检索部分正在进行中，当前集成 LlamaIndex 向量存储适配器，采用余弦相似度实现 Top-K 段落召回，向量索引配置与检索性能调优尚在调试阶段。

## 5.11 当前风险识别与下阶段计划

在风险管理上，我们识别出三项需要重点关注的风险。

第一项是 SSE 代理兼容性风险。Nginx 反向代理下 SSE 长连接配置较为复杂，可能影响流式内容的实时传输。为了应对该风险，我们准备采用基于 HTTP 轮询的降级备用方案，在极端情况下实现快速切换，保障核心功能可用性。

第二项是 pgvector 调试周期风险。向量维度匹配与检索性能调优的耗时存在不确定性。我们计划在调试周期超出预期时，先以简单余弦相似度计算过渡，优先打通检索链路的功能可用性，在后续再继续迭代优化。

第三项是 Session Service 实现复杂度风险。状态机路由与异步任务编排逻辑涉及多个组件的协调，实现复杂度较高。因此，我们计划优先聚焦引导式模式的完整主流程，先打通端到端可演示能力，再逐步扩展至完整功能集。

在 Sprint 2 计划上，执行周期为 2026 年 4 月 13 日至 2026 年 4 月 26 日，主要目标有四项。
第一，DeepSearch 模块，完成网络搜索 API 接入，实现多轮查询改写与跨源信息聚合的基础流程。
